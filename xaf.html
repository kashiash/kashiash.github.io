To jest decyzja numer jeden. Jeśli nie wiesz, od czego zacząć, najpierw sprawdź Supported XAF UI Platforms, a potem sekcję Storage, ORM, and Business Model Design. Dopiero później schodź w kontrolery, edytory i szczegóły UI.

Mój skrót

XAF daje szybko UI, ale płacisz zrozumieniem mechaniki.

Jeśli rozumiesz ObjectSpace, model biznesowy, kontrolery i security, to framework naprawdę przyspiesza. Jeśli nie, projekt szybko robi się „magiczny” i trudny w utrzymaniu.

Skrót

Co warto wiedzieć bez czytania wszystkiego naraz

1. XAF to nie tylko UI generator

XAF buduje aplikacje biznesowe wokół modelu biznesowego, Application Model, ObjectSpace i modułów. UI jest efektem tych decyzji, a nie osobnym światem.

2. ORM i model biznesowy są fundamentem

  • Do wyboru masz EF Core albo XPO.
  • Ta decyzja wpływa na business classes, ObjectSpace i dalszą organizację kodu.
  • Jeśli dane nie pochodzą z ORM, używaj non-persistent objects.

3. Kontrolery i akcje to miejsce, gdzie robisz realną logikę UI

  • Kontrolery spajają zachowanie widoków, reakcji na stan i logikę operacyjną.
  • Actions to menu commands i workflow użytkownika.
  • To jest sekcja, do której wraca się bardzo często przy customizacji.

4. Web API nie jest dodatkiem „na potem”

  • XAF ma gotowy backend oparty o ASP.NET Core OData.
  • Możesz hostować go razem z Blazor Server albo jako osobny projekt.
  • To dobry kierunek, jeśli chcesz obsłużyć MAUI, JS albo inne klienty poza XAF UI.

5. Bez security i validation projekt jest niepełny

  • Security w XAF to nie kosmetyka, tylko rdzeń aplikacji biznesowej.
  • Validation i Conditional Appearance szybko porządkują zachowanie formularzy.
  • Deployment warto czytać wcześniej, nie dopiero przed publikacją.
